# 任务面板完整排查、问题清单与优化方案

## 目标

对任务面板的服务链路、状态判定、渲染呈现和交互闭环做一次完整审计，明确当前架构问题、潜在 bug、逻辑不合理点，并给出可执行的修复与重构方案。

## 排查范围

- 终端捕获与会话聚合：`TerminalCaptureService`
- 会话任务判定与快照生成：`TaskSessionEngine`、`TaskSessionStateMachine`
- 文本标准化与交互菜单提取：`TaskSessionTextToolkit`
- 策略装载与 JSON 规则：`ConfigurableTaskSessionStrategy`、`TaskStrategies/*.json`
- 任务面板 UI 与交互：`IslandView`、`TaskPanelFeatureView`
- 相关测试：`Tests/BulingIslandTests/*Task*`

## 现状结论

### 1. 任务面板核心能力已具备，但职责边界不清晰

当前链路已经具备以下能力：

- 多终端会话统一捕获
- 基于策略的任务生命周期识别
- 双行摘要渲染
- 等待确认场景的结构化 prompt 解析
- pill 状态与任务面板复用同一套策略分析

但这些能力没有被明确拆分为“采集层 / 判定层 / 面板展示层 / 交互执行层”四层，导致：

- `IslandView` 直接承担大量任务面板状态、排序、拖拽、交互分发逻辑
- 结构化交互虽然已能解析，但此前未完全接通到面板 UI
- 会话生命周期稳定器与 UI 刷新耦合较深，不利于单元测试和后续扩展

### 2. 已确认存在的 bug / 隐患

#### P0: 状态机粘滞状态只稳定一轮

位置：`TaskSessionStateMachine`

问题：

- 原实现把 `proposed` 写回缓存，而不是把 `stabilized` 写回缓存。
- 结果是 `error` / `success` 的稳定窗口只在下一次轮询生效一次。
- 这会导致刚完成或刚报错的任务在后续轮询中过早掉回 `idle`，面板状态闪烁。

影响：

- 任务完成态和错误态稳定性不足
- pill 和任务面板会出现视觉抖动

处理：

- 本次已修复为写回稳定后的生命周期，并补充重复轮询测试

#### P1: 等待确认的结构化交互解析已存在，但面板没有提供可点击操作

位置：`TaskSessionTextToolkit`、`IslandView`

问题：

- 系统已能提取 `interactionPrompt`、`options`、`confirmButton`、`controlButtons`
- 但原任务面板仅提示“请点击任务前往终端处理”，没有把这些结构化操作渲染出来
- 等于后端解析和前端呈现脱节，形成“半截功能”

影响：

- 用户必须离开面板切回终端处理确认
- 导致任务面板在“等待输入”这个最需要操作的场景下可用性最低

处理：

- 本次将等待确认场景渲染为可点击按钮区
- 保留“前往终端”兜底入口

#### P1: 会话跟踪状态未及时回收，存在陈旧状态残留

位置：`TerminalCaptureService`

问题：

- `sessionDigest`、`sessionLastChangedAt`、错误确认缓存会在会话消失后继续保留
- 这会让服务层保留无效会话的历史状态

影响：

- 长时间运行后存在状态污染与不必要内存增长风险
- 某些基于“最近活跃”的判断会受到已关闭会话影响

处理：

- 本次在消费新一轮捕获结果前回收已消失会话的跟踪状态

#### P2: 任务面板交互状态有存储但未形成完整闭环

位置：`IslandView`

问题：

- `taskInteractionSelections` 已存在，但原先没有被渲染消费
- 会话消失后也没有清理这部分状态

影响：

- 逻辑显得不完整，容易留下脏状态

处理：

- 本次已将其接入按钮选中态，并在会话回收时同步清理

### 3. 架构问题

#### `IslandView` 过载

当前任务面板相关逻辑大量堆叠在 `IslandView` 内，包括：

- 分组
- 排序
- 拖拽
- pin/mute
- 等待确认的展示
- 交互事件分发
- 外部终端跳转

这会带来几个问题：

- 修改任务面板容易波及 Claude 面板和整个岛视图
- 很难对任务面板做独立 UI 测试
- 逻辑重复积累后会继续变成“大型宿主 View”

建议的重构方向：

- 抽出 `TaskPanelView` 作为独立主视图
- 抽出 `TaskPanelViewState` / `TaskPanelCoordinator` 统一管理排序、分组、交互状态
- `IslandView` 只负责容器切换，不再关心任务面板内部实现细节

#### 任务生命周期判定过于依赖字符串启发式

虽然 JSON 策略让系统具备了可配置性，但当前仍主要依赖尾部文本关键字：

- 优点：快速、灵活、低成本
- 问题：容易被不同终端主题、输出格式、CLI 版本变化影响

建议的中期改造：

- 保持 JSON 策略作为第一层兼容方案
- 增加“结构化 signal 适配层”，让支持的工具优先输出明确状态
- 将关键词匹配降级为 fallback，而不是主路径

## 本次已实施修改

### 代码修复

- 修复 `TaskSessionStateMachine` 粘滞状态回写错误
- 为等待确认场景新增 `TaskInteractionPromptSection`
- 任务面板现在可直接展示并触发选项、确认按钮、控制按钮
- 为等待确认场景保留“前往终端”兜底操作
- 多选菜单在面板内操作时，优先复用结构化 action；发送失败时回退到激活终端
- 会话回收时同步清理终端捕获缓存和任务交互选择状态

### 测试补强

- 新增状态机“重复 idle 轮询仍保持 sticky”测试
- 覆盖 error / success 两类稳定窗口

## 后续建议

### 第一阶段

- 继续把任务面板从 `IslandView` 中拆出，至少形成独立 `TaskPanelView`
- 将分组、排序、拖拽状态封装到独立状态对象

### 第二阶段

- 为等待确认按钮交互增加更直接的单元测试或快照测试
- 对 `TaskInteractionPrompt` 的导航型菜单增加更多 fixture 覆盖

### 第三阶段

- 为 Codex / Claude / 通用终端任务引入结构化状态适配协议
- 让策略 JSON 负责兜底，不再承担全部生命周期判断职责

## 结论

当前任务面板不是“推倒重来”的状态，但已经出现典型的宿主 View 过载和能力半接通问题。短期内通过修 bug、把交互能力接回 UI、回收陈旧状态，可以显著提升稳定性和可用性；中期应继续把任务面板从 `IslandView` 拆出，形成可测试、可演进的独立模块。
